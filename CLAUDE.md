# agent-squad

Multi-agent orchestration system for Claude: a human-facing orchestrator
spawns delegated worker agents in isolated tmux sessions, tracks their work
via a local ticket system, and monitors them through a live dashboard.

## Architecture

- **Orchestrator**: the human-facing Claude session. Breaks work into
  stories, spawns workers, consolidates and ships their results.
- **Workers**: delegated Claude sessions (`tmux` sessions prefixed
  `claude-`), each working a single story/ticket in isolation.
- **Dashboard**: `workers-follow-up/` — read-only web UI over the tmux
  sessions and ticket state, served on `http://127.0.0.1:8787/`.

## Components

- `./squad` → wrapper for `claude-cli/main.py`: spawn/capture/kill worker
  tmux sessions. Worker model defaults to `sonnet` (`--model` flag >
  `SQUAD_MODEL` env var > default), so workers never inherit an
  expensive default model.
- `./context` → wrapper for `context-cli/main.py`: manages worker
  **roles** (`context-cli/roles/*.yaml`, e.g. `worker`, `orchestrator`)
  and reusable **directives** (`context-cli/directives/*.yaml`, e.g.
  `base`, `code-quality`) that get composed into a worker's prompt.
- `./tickets` → wrapper for `tickets-cli/main.py`: local ticket tracking
  for work delegated to workers (create/assign/update/comment/list).
- `workers-follow-up/` → the monitoring dashboard (stdlib-only
  `server.py` using `http.server.SimpleHTTPRequestHandler` +
  `ThreadingHTTPServer` + a small JS app), installable as a macOS
  LaunchAgent.

Each CLI is a standalone `uv`-managed Python project (own
`pyproject.toml`/`uv.lock`); the root wrapper scripts (`claude`,
`context`, `squad`, `tickets`) just `cd` into the right subdir and
`exec uv run python main.py "$@"`.

## Worker conventions

- Each spawned worker gets a per-session prompt file at
  `.squad-prompts/<session>.txt` (role + directives + task, generated by
  `context-cli` and appended with the task body) and a launch script at
  `.squad-runs/<session>.sh`. Both are gitignored (runtime artifacts).
- Workers report blockers or questions via `/squad:waiting`, tracked in
  `signals/waiting/` and `signals/responses/` until the orchestrator
  answers with `/squad:respond`.
- Workers **never** commit, push, or merge. They finish their ticket
  (`./tickets update <id> --status done`), leave a summary comment, and
  stop. The orchestrator alone consolidates changes into clean commits,
  pushes, and opens PRs.

## Workflow

One Linear story = one ticket = one worker session. No change lands
without a story/ticket behind it, even a trivial one. Complex changes
get split into several independent stories rather than one large one.

Ticket lifecycle: worker session opens → ticket **in-progress** → worker
finishes and sets the ticket **done** (`./tickets update <id> --status
done`) → orchestrator runs a review/control pass → in Linear, the
matching issue moves In Progress → In Review → Done on pass, or back to
In Progress with review notes on fail. (`in-review` is a Linear issue
state, not a local ticket status — `tickets-cli` only accepts
open/in-progress/blocked/waiting/done.)

## Verification

Sanity-check the three CLIs compile after any change to them:

```bash
python -m compileall claude-cli context-cli tickets-cli
```

None of the three CLIs has its own tests; the only tests in the repo are
`workers-follow-up/providers/test_*.py` for the dashboard.
